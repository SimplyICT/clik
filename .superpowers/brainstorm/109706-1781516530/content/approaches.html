Two approaches

Option A: Extend Current Stack Recommended

Add columns to existing assets table, add asset_id to requests, build mobile views, new parts inventory table.

Pros

  • Reuses existing request lifecycle, auth, notifications
  • Parts = clean new table, manageable scope
  • QR codes = URL-based, no hardware
  • Custom fields = JSONB, no schema migrations

Cons

  • Assets page needs full rewrite
  • Mobile needs new views
  • Parts inventory is net-new

Option B: Standalone Asset Management Service

Dedicated Python service or module with its own DB tables. Separate from the existing assets CRUD entirely.

Pros

  • Clean break from old CRUD
  • Independent schema evolution
  • Could become a standalone product

Cons

  • Much more work (auth duplication, sync, separate deployment)
  • Jobs still need to integrate with request system
  • Overkill for this stage
